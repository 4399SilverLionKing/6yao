# 六爻正式排盘第二轮设计

## 1. 目标

本轮目标是在现有首版工具站基础上，将当前“可运行的结构化排盘”升级为《增删卜易》口径下的正式排盘。

本轮只解决正式盘面生成问题，不进入断卦判断层。

本轮完成后，系统需要能够根据：

- 三枚铜钱起卦结果
- 用户输入的公历时间
- 固定北京时间口径

自动生成包含以下信息的正式排盘结果：

- 本卦
- 变卦
- 纳甲
- 六亲
- 六神
- 世应
- 伏神
- 月建
- 日辰
- 旬空

## 2. 本轮边界

### 2.1 本轮包含

- 保留三枚铜钱起卦
- 新增公历日期时间输入
- 固定以 `Asia/Shanghai` 为排盘时区
- 自动换算月建、日辰、旬空
- 用《增删卜易》口径输出正式装盘结果
- 用更严格的规则测试和整盘样例校验替换当前简化逻辑

### 2.2 本轮不包含

- 用神判定
- 元神、忌神、仇神
- 旺衰、生克、冲合、墓绝
- 应期判断
- 断语生成
- AI 解卦

也就是说，本轮目标是“盘面可信”，不是“断卦完成”。

## 3. 选型结论

本轮推荐技术方案：

- 历法换算：`lunar-typescript`
- 六爻规则：本地静态规则表
- 输出层：继续沿用 React + TypeScript 前端本地计算

采用 `lunar-typescript` 的原因：

- 能直接提供干支相关能力
- 文档包含旬与旬空相关接口
- TypeScript 友好，减少现有项目里的类型摩擦
- 适合做本地计算，不需要依赖远程接口

本轮仍然保持纯前端实现，不新增后端服务。

## 4. 规则口径

本轮正式排盘统一按《增删卜易》口径实现，并固定以下约束：

- 起卦时间以北京时间计算
- 月建、日辰、旬空由系统自动换算
- 六神按日干排布
- 纳甲、世应、伏神、六亲均使用同一套固定规则表

本轮不允许：

- 在组件中临时写规则判断
- 同时混用不同流派的规则口径
- 对不同卦单独写硬编码例外

所有规则差异必须落在 `src/data/zengshan/` 的静态数据中。

## 5. 架构调整

### 5.1 新增模块边界

建议新增三组模块：

```text
src/
  core/
    calendar/
    liuyao/
    formatter/
  data/
    zengshan/
```

职责划分如下：

- `src/core/calendar/`
  - 处理公历时间输入
  - 固定北京时间
  - 输出月建、日辰、旬空等时间盘面信息
- `src/core/liuyao/`
  - 处理纳甲、六亲、六神、世应、伏神
  - 只接收标准化卦象和时间上下文
- `src/data/zengshan/`
  - 存放《增删卜易》口径下的静态规则表
  - 包括八卦纳甲、世应规则、伏神映射、六神起例等

### 5.2 现有模块调整

现有 `src/core/relations/` 中的简化规则不应继续扩写，应逐步替换为正式规则层：

- 简化版六神逻辑替换为依赖日干的正式规则
- 简化版六亲逻辑替换为依赖纳甲和五行的正式规则
- 当前 `shiYing` 的简化判断替换为正式世应规则表

`src/core/formatter/` 继续保留，但职责改为：

- 聚合历法结果
- 聚合正式排盘结果
- 输出 UI 可消费的统一结构

## 6. 输入与数据流

### 6.1 输入结构

本轮输入需要包含两部分：

- 六次铜钱结果
- 公历日期时间

用户输入时间时，不暴露时区选择。系统默认按北京时间解释和换算。

### 6.2 新数据流

建议正式排盘采用如下数据流：

`用户输入 -> 标准化起卦 -> 本卦/变卦 -> 北京时间历法换算 -> 月建/日辰/旬空 -> 纳甲 -> 六亲/六神/世应/伏神 -> 格式化结果对象 -> UI 渲染`

关键原则：

- 历法换算与六爻规则分离
- 六爻规则层只消费明确的“时间上下文对象”
- 最终统一输出一个正式排盘结果对象

## 7. 正式排盘对象

建议扩展现有结果对象，至少包含以下字段：

- `originalHexagram`
- `changedHexagram`
- `movingLines`
- `calendar`
- `monthBranch`
- `dayGanzhi`
- `dayXunKong`
- `lines`
- `hiddenLines`
- `sixRelations`
- `sixSpirits`
- `shiYing`
- `meta`

其中：

- `calendar`
  - 保存用户输入时间、北京时间、农历换算结果
- `lines`
  - 每一爻都包含纳甲、五行、六亲、六神、世应标记
- `hiddenLines`
  - 伏神信息单独输出，避免和主爻结构混杂

这样做的价值：

- 页面组件不再自行拼装盘面信息
- 后续新增导出、打印、AI 解读时可直接复用
- 测试可以围绕完整盘对象做整盘快照校验

## 8. 规则数据设计

本轮的核心不在算法技巧，而在规则数据的清晰组织。

建议在 `src/data/zengshan/` 中拆出以下数据文件：

- `naJia.ts`
  - 八卦纳甲与各爻地支映射
- `shiYing.ts`
  - 世应规则表
- `fuShen.ts`
  - 伏神规则表
- `sixSpirits.ts`
  - 六神起例表，按日干确定起始顺序
- `relations.ts`
  - 六亲判定基础规则

要求：

- 所有表必须可读、可测试
- 表驱动优先，不把规则散落在函数分支里
- 静态表命名和结构要直接反映排盘含义

## 9. 页面调整

### 9.1 起卦页

起卦页需要新增：

- 公历日期时间输入
- 默认北京时间说明

用户不需要手动填月建、日辰、旬空，系统自动生成。

### 9.2 结果页

结果页需要新增两个信息区：

- 时间盘面区
  - 公历时间
  - 农历信息
  - 月建
  - 日辰
  - 旬空
- 正式装盘区
  - 每爻纳甲
  - 六亲
  - 六神
  - 世应标记
  - 伏神信息

结果页应明确区分：

- 主卦结构
- 变卦结构
- 时间信息
- 伏神附加信息

避免把所有信息堆到一张表里导致可读性下降。

## 10. 错误处理

本轮错误处理需要覆盖新的时间维度。

至少处理以下场景：

- 时间未填写
- 时间格式非法
- 历法换算失败
- 月建或日辰缺失
- 纳甲规则查表失败
- 世应或伏神规则不完整

策略要求：

- 时间输入不完整时阻止提交
- 历法换算失败时给出明确错误提示
- 规则表缺项时抛出显式错误，不能静默回退
- 结果页不展示部分正确、部分缺失的正式盘

## 11. 测试策略

本轮测试强度必须高于首版。

### 11.1 历法测试

固定若干北京时间样例，校验：

- 月建
- 日辰
- 旬空

测试应包含边界样例，例如：

- 节气附近日期
- 月份切换附近日期
- 同一天不同时间样例

### 11.2 规则测试

固定卦例，校验：

- 纳甲
- 六亲
- 六神
- 世应
- 伏神

每个模块都应有独立测试，避免整盘失败后无法定位问题。

### 11.3 整盘样例测试

加入若干整盘回归测试，直接校验完整结果对象中的关键字段：

- 时间信息
- 六爻主盘
- 伏神
- 世应
- 动爻
- 变卦

这类测试的作用是防止后续规则重构破坏整体盘面。

## 12. 风险与约束

本轮主要风险有三类：

- 历法口径错误
- 规则表不完整
- 不同流派规则混入

控制方式：

- 历法统一依赖单一库
- 规则统一落在本地表中
- 所有样例测试都明确注明《增删卜易》口径

此外必须接受一个现实约束：

即使完成本轮，也只是“正式排盘可信”，仍不代表“断卦能力完成”。

## 13. 最终结论

第二轮应作为一个独立子项目推进，目标是把当前简化盘面升级为《增删卜易》口径的正式排盘层。

实施重点不是继续堆 UI，而是补齐三块基础能力：

- 北京时间历法换算
- 《增删卜易》规则表
- 更严格的样例测试

完成后，网站将从“可运行的排盘工具”升级为“盘面可信的正式排盘工具”，为后续断卦层打下稳定基础。
