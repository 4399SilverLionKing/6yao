# 六爻在线起卦与自动排盘首版设计

## 1. 目标

本项目首版是一个六爻在线工具站，核心目标是让用户可以在网页上完成起卦，并立即得到结构化排盘结果。

本阶段只解决两个问题：

- 在线起卦
- 自动排盘

本阶段明确不做以下能力：

- 用户登录
- 历史记录
- 收藏与分享体系
- 会员与付费
- AI 解卦
- 后端服务与数据库

这样做的目的是压缩首版范围，优先验证工具流程是否顺畅、排盘结果是否稳定正确。

## 2. 选型结论

首版采用以下技术栈：

- Vite
- React
- TypeScript
- Tailwind CSS
- Vitest
- Cloudflare Pages

部署形态为纯静态站点。所有起卦、排盘和结果整理逻辑均在前端本地完成，构建产物为 `dist`，直接部署到 Cloudflare Pages。

选择这套方案的原因：

- 上线速度快，没有服务端与数据库配置成本
- 维护简单，首版复杂度最低
- Cloudflare Pages 免费静态托管足够承载当前需求
- 后续如果接入 AI、保存记录或轻量接口，可以在此基础上补 Workers 层，不需要推翻前端结构

## 3. 产品边界

### 3.1 首版包含

- 首页说明与入口
- 起卦流程页
- 自动排盘结果页
- 三枚铜钱起卦
- 手动录入投掷结果
- 一键随机生成投掷结果
- 本卦、变卦、动爻展示
- 六亲、六神、世应等结构化排盘展示

### 3.2 首版不包含

- 多种起卦方式并行支持
- 用户中心
- 云端存储
- 富文本内容系统
- 咨询平台能力
- 支付系统

## 4. 架构设计

### 4.1 总体结构

系统采用前端单体应用结构：

`用户输入 -> 起卦标准化 -> 排盘计算 -> 结果格式化 -> 页面渲染`

架构目标：

- 页面层只负责输入与展示
- 核心排盘逻辑集中在独立模块
- 静态数据与规则运算分离
- 输出统一结果对象，避免 UI 与算法耦合

### 4.2 目录建议

```text
src/
  pages/
  components/
  core/
    divination/
    hexagram/
    relations/
    formatter/
  data/
  types/
  tests/
```

模块职责如下：

- `pages/`
  - 页面级视图
  - 组织首页、起卦页、结果页
- `components/`
  - 纯展示组件
  - 如卦象卡片、爻位表、结果摘要等
- `core/divination/`
  - 处理起卦输入
  - 将手动录入和随机生成统一成标准结构
- `core/hexagram/`
  - 负责本卦、变卦、动爻计算
- `core/relations/`
  - 负责六亲、六神、世应等规则计算
- `core/formatter/`
  - 将底层计算结果整理成页面可直接消费的数据对象
- `data/`
  - 存放六爻静态规则数据和映射表
- `types/`
  - 定义输入、卦象、爻、排盘结果等类型
- `tests/`
  - 以核心规则回归测试为主

## 5. 页面设计

### 5.1 首页 `/`

职责：

- 说明这是一个六爻在线工具
- 说明首版支持的起卦方式
- 提供进入起卦流程的入口

首页不承担内容站职责，不做复杂 SEO 策略，不承载大量文章和落地页。

### 5.2 起卦页 `/cast`

职责：

- 输入占问内容
- 选择三枚铜钱起卦
- 支持手动录入六次投掷结果
- 支持一键随机生成六次结果
- 输入完成后触发排盘

页面重点是流程清晰，不做复杂动画和附加功能。

### 5.3 结果页 `/result`

职责：

- 展示本卦
- 展示变卦
- 标记动爻
- 展示六爻结构
- 展示六亲、六神、世应
- 展示基础元信息

结果页只展示结构化排盘，不在首版承担深度解卦解释。

## 6. 起卦方式设计

首版只支持一种起卦方式：

- 三枚铜钱起卦

原因：

- 用户认知成本最低
- 规则明确，适合前端本地纯计算
- 可以先稳定起卦与排盘主流程
- 后续扩展时间起卦、数字起卦时，不需要推翻现有结构

前端交互上支持两种输入形式：

- 手动录入每次投掷结果
- 自动随机生成每次投掷结果

但底层统一映射为同一套标准输入数据结构，保证后续计算链路不因输入来源不同而分叉。

## 7. 数据流与核心对象

### 7.1 数据流

推荐采用以下固定数据流：

`用户输入 -> 标准化起卦结果 -> 本卦/动爻/变卦计算 -> 六亲/六神/世应计算 -> 格式化结果对象 -> UI 渲染`

其中：

- 用户输入可以来自手动录入或随机生成
- 标准化步骤将不同输入方式转换为统一的爻数据
- 排盘规则层只接收标准化数据
- 页面只消费格式化后的排盘结果

### 7.2 统一结果对象

排盘完成后，建议输出统一结果对象，字段可包括：

- `originalHexagram`
- `changedHexagram`
- `movingLines`
- `lines`
- `sixRelations`
- `sixSpirits`
- `shiYing`
- `meta`

这样做的价值：

- UI 组件只依赖统一对象
- 未来新增导出、分享、AI 解读时可直接复用
- 排盘结果可用于测试快照和回归校验

## 8. 静态数据策略

首版所有规则数据都采用本地静态数据管理，不接数据库。

建议放入 `src/data/` 的内容包括：

- 八卦基础定义
- 六十四卦基础映射
- 爻位定义
- 六神顺序规则
- 六亲映射关系
- 世应规则数据
- 卦名与基础描述信息

静态数据与计算逻辑分离，避免在算法函数内部硬编码大段规则。

## 9. 状态管理策略

首版不引入全局状态库。

建议采用：

- React 组件状态
- 必要时使用少量 Context 传递起卦结果

原因：

- 页面数量少
- 流程简单
- 引入额外状态库只会增加学习与维护成本

当后续新增用户记录、复杂筛选、多页共享状态时，再评估状态管理升级。

## 10. 错误处理

首版错误处理采用轻量但明确的策略。

需要覆盖的场景：

- 用户输入不完整
- 输入结果非法
- 起卦数据无法标准化
- 排盘计算出现未覆盖规则
- 结果页在无数据状态下被直接访问

建议处理方式：

- 输入不完整时禁止提交，并给出明确提示
- 非法输入立即在当前页面提示，不进入排盘流程
- 计算异常时展示统一错误卡片，而不是渲染半成品结果
- 对结果页增加守卫逻辑，无有效排盘结果时引导用户返回重新起卦

## 11. 测试策略

测试重点放在 `src/core/`，而不是页面层。

### 11.1 核心测试内容

- 三枚铜钱结果到阴阳、老阴、老阳、少阴、少阳的映射
- 六爻组合到本卦的计算
- 动爻到变卦的计算
- 六亲规则输出
- 六神规则输出
- 世应定位规则输出
- 固定样例盘的完整回归测试

### 11.2 页面测试内容

页面只覆盖关键流程：

- 起卦输入是否可完成
- 排盘结果是否可展示
- 非法状态是否有正确提示

测试比例建议：

- 主要精力放在核心规则测试
- 页面流程测试保持轻量

## 12. 非功能要求

首版需满足以下要求：

- 移动端可用
- 页面加载快
- 输入与结果展示清晰
- 核心规则层可测试、可维护
- 部署过程简单稳定

由于首版是工具站，不追求复杂视觉设计，优先保证流程和结果可信度。

## 13. 后续扩展方向

首版完成后，可沿以下方向增量扩展：

- 增加更多起卦方式
- 增加排盘结果导出与分享
- 接入 AI 解卦
- 增加历史记录和账户体系
- 增加内容页和 SEO 页面
- 接入 Cloudflare Workers、D1、R2 等后端能力

当前架构允许这些能力逐步增加，但不应在首版提前引入。

## 14. 最终结论

首版应当以最小可用工具为目标，采用纯前端静态站形态，优先完成三枚铜钱起卦与自动排盘的稳定闭环。技术栈使用 Vite、React、TypeScript、Tailwind CSS、Vitest，并部署到 Cloudflare Pages。

这套方案满足当前目标：

- 开发快
- 成本低
- 维护轻
- 易于验证核心规则正确性
- 后续扩展空间清晰
